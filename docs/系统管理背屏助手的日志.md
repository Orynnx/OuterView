# 系统管理背屏助手的日志

## 结论

系统卡片通常由真实 Android 通知驱动，但通知不是 MAML loader 的必要组成。真正需要的是完整执行通知解析后的宿主原生 Post Runnable，使 business、manager list、模板路径和 MAML loader 按原生顺序工作。仅修改 manager list 或 `notification_widget.json` 都不是可靠的显示入口。

OuterView v3 使用以下链路：

```text
导入 Widget v2 ZIP
-> App 私有目录保存
-> Hook 在 com.xiaomi.subscreencenter 进程中部署模板
-> App 通过 Host API activateCard 发送 rear/focus payload
-> Hook 在宿主主线程构造并执行原生 Post Runnable
-> manager list
-> 小米原生 MAML loader
```

隐藏卡片通过 Host API `deactivateCard` 调用宿主按 package/business 移除方法完成。OuterView 不申请通知权限、不发布 Android 通知，也不直接读写系统 `notification_widget.json`。

## 系统路径

当前设备用户 0 的主要路径：

```text
/data/system/theme_magic/users/0/subscreencenter/smart_assistant/
/data/system/theme_magic/users/0/subscreencenter/notification/notification_widget.json
/data/system/theme_magic/users/0/subscreencenter/config/user_pref.json
/data/system/theme_magic/users/0/subscreencenter/config/widget.json
/data/system/theme_magic/users/0/subscreencenter/logs/app.log
```

`widget.json` 保存普通背屏主题组件；`notification_widget.json` 保存 Smart Assistant 通知卡片的持久化结果，两者不是同一种卡片。

## 系统模板格式

`smart_assistant` 下的系统模板是无扩展名 ZIP 文件，不是解压目录。例如：

```text
alarm
car_hailing
ev
food_delivery
miHomeCamera
music
phone
privacy
sports_schedule
stock
timer
```

系统模板拥有正确的宿主 UID、SELinux context，并包含顶层 `manifest.xml`、资源和多语言文件。典型根节点为：

```xml
<Widget frameRate="0" scaleByDensity="false" screenWidth="1080" version="2">
```

旧版 10 个 FunCards 只有约 1 KB 的静态 `manifest.xml`，根节点为 `<Root>`，不能作为 Smart Assistant Widget v2 的兼容性证据。

## 原生通知链路研究证据

2026-07-10 10:11，系统隐身模式开关产生了完整的通知生命周期。

显示时日志包含：

```text
postNotification: Package=com.miui.securitycenter, NotificationId=20000001
handleNotificationPosted
Using compositeKey: com.miui.securitycenter:20000001 (business: privacy)
MamlWidget - onCreate path = .../smart_assistant/privacy
MamlWidget - Load ... loaded: true
```

隐藏时日志包含：

```text
Widget removed: com.miui.securitycenter:20000001 from index 0
removeNotification
handleNotificationRemoved
```

这说明 packageName、notificationId 和 business 共同决定原生卡片身份。OuterView v3 使用独立合成 runtime ID 和相同 extras 结构调用通知解析后的宿主入口，不创建系统通知。

通知 extras 至少需要业务参数：

```json
{
  "miui.rear.param": {
    "rear_param_v1": {
      "business": "privacy",
      "priority": 700,
      "maml_config": {}
    }
  }
}
```

部分模板读取 `miui.focus.param`、`data_str` 或业务专用 `maml_config`，因此“模板加载成功”和“内容数据完整”应分别诊断。

## 已确认的失败模式

### 路径不存在

旧模块 `com.example.codexpanel` 曾反复发布：

```text
business=codex_panel
path=.../smart_assistant/re_codex_panel
fileExists=false
```

宿主随后创建空白卡片。force-stop 模块 App 无效，因为 Xposed 代码运行在宿主进程中；必须禁用对应模块或移除其宿主注入。

### 多个 Hook 服务竞争

旧 REAREye、旧 CodexPanel 和 FunCards Tester 曾同时注入 `com.xiaomi.subscreencenter`。它们重复 Hook manager 并响应同一 Binder bootstrap，导致客户端可能连接到不含新 AIDL transaction 的旧服务，表现为：

```text
Hook service returned an empty result
```

v2 使用独立 action、`apiVersion=2`、providerPackage 和 providerInstanceId 校验，不再接受旧服务。

### 持久化文件权限错误

实验恢复 `notification_widget.json` 时曾设置为：

```text
system:system 0660
```

但宿主数据属于 `com.xiaomi.subscreencenter` 的 UID。宿主因此记录：

```text
File .../notification/notification_widget.json load failed
```

管理器 v2 永远不修改该文件的内容、owner、mode 或 SELinux context。

### 异步操作提前返回成功

旧实现把 manager 删除放入 `Handler.post` 后立即返回成功，导致 UI 状态与宿主实际状态不一致。v2 在显示和隐藏后最多轮询 5 秒，只有 manager list 和 live widget 达到目标状态才返回成功。

## v2 管理边界

自定义模板只允许写入：

```text
/data/system/theme_magic/users/<userId>/subscreencenter/smart_assistant/reareye_custom_<32位cardId>
```

宿主 registry 只允许写入：

```text
/data/system/theme_magic/users/<userId>/subscreencenter/reareye_custom_cards/registry.json
```

删除前必须同时满足：目标是 `smart_assistant` 的直接子文件、文件名以 `reareye_custom_` 开头、后缀是合法 32 位 cardId。系统模板、其他模块模板和系统持久化文件不属于管理范围。
